Hola de nuevo, bienvenidos al segundo punto del curso de MySQL del Master Desero de Bixeo, en el cual hablaremos de las relaciones de las bases de datos. Como ya hemos comentado anteriormente, una entidad es cualquier objeto real o abstracto del cual se puede almacenar información en una base de datos. Por ejemplo, una base de datos se puede tener la entidad ciudad. Una relación es la representación de asociaciones entre entidades, permite relacionar datos entre sí. En este ejemplo que vemos en pantalla tenemos dos entidades, la entidad ciudad y la entidad país y tenemos una relación que a través de un campo estas dos tablas están relacionadas. Cada ciudad tiene su país y cada país tiene sus ciudades. Estamos hablando en este caso de una relación de uno a varios. Un país tiene varias ciudades pero una ciudad únicamente pertenece a un país. Las entidades y sus instancias tienen que ser distinguibles del resto. Para eso usamos el identificador o llave, que es un conjunto de uno o más atributos cuyos valores son únicos en cada instancia de la entidad. Para que un atributo pueda seguir identificador debe cumplir los siguientes requisitos. Tiene que distinguir cada ejemplar del antidato relación. Todos los ejemplares de una entidad debe tener el mismo identificador. Un identificador puede estar formado por más de un atributo. Pueden haber varios identificadores candidatos. Escogeremos el de más importancia del sistema. El resto pasan a ser identificadores alternativos. En el caso de la entidad persona un NIF cumple todos los requisitos anteriores. Muchas veces se opta por generar identificadores autoincrementales. Si volvemos un segundo al ejemplo anterior vemos que tanto en las tablas de ciudad como país el identificador es un campo autoincremental. Esto significa que cuando introduzcamos el primer registro se la asignará el ID1 al segundo registro el ID2 y así sucesivamente. En el caso de que se elimine un registro este número nunca más aparecerá porque como es autoincremental se irá sumando uno cada vez que se añade un registro en ambas tablas. En la tabla país este identificador de país que funciona como clave principal como llave no se podrá repetir. En cambio en la tabla ciudad el campo country ID sí que se puede repetir porque en esta tabla tendremos varias ciudades que pertenezcan a un único país. Por lo tanto este campo en esta tabla sí que se repetirá pero el city ID no se repetirá en la tabla ciudades porque aquí también funcionará como clave este dato. Dentro de los diferentes tipos de relaciones vamos a empezar por la relación de uno a uno. Supongamos que tenemos una tabla para personas y otra para direcciones. No es muy común ya que se podría incluir toda la información de la misma tabla, vale al final la dirección podría estar dentro de la tabla persona como un atributo más. Pero bueno si seccionamos la dirección en tipos y es calle, si es vía, el nombre, el número, el código postal, el país, etcétera, etcétera pues a lo mejor no nos interesa por la estructura que tenemos que la tabla persona tenga tantos atributos, tenga tantos campos y preferimos dejar todas las direcciones en una tabla aparte. Para poder establecer esta relación es necesario crear el campo y de dirección en la tabla persona. Este campo recibe el nombre de clave foránea. El campo que en la tabla dirección es el campo llave o clave primaria y de dirección en la tabla persona funcionaría como clave foránea. En este caso las reacciones de uno a uno porque en principio una persona únicamente tiene una dirección y una dirección únicamente está asociada a una persona. El segundo tipo de relación que vamos a ver es la relación uno a varios que es la misma que hemos comentado anteriormente con el ejemplo de ciudades y países. Supongamos que tenemos una tabla para clientes y otra para pedidos. Un cliente puede realizar varios pedidos y a su vez un pedido únicamente pertenece a un cliente sería el mismo caso que con las ciudades. Y aquí vemos los ejemplos de una tabla cliente con el identificador autoincremental y en la siguiente tabla que es la tabla pedido pues asociamos a cada uno de los pedidos que vayan entrando a que cliente pertenecen incluyendo el campo y de cliente dentro de esta tabla al igual que en el caso anterior el campo y de cliente de la tabla pedido estará funcionando como clave foráneo y el campo y de la tabla cliente y el campo y de pedido de la tabla pedido estarán funcionando como claves principales. Para generar este tipo de relaciones es necesario crear una nueva entidad.
